如果说 OpenCode 已经像一个愿意下场干活的终端搭子，那 Oh My OpenCode（下面我就偷懒叫它 OMO）更像是给这个搭子配了一整个施工队：有负责总调度的，有负责查资料的，有负责深挖代码的，还有那种专门在你快要跑偏时把你拽回来的“监工型同事”。

它最吸引我的地方，不是“agent 更多了”这句宣传词本身，而是它把很多原本要靠人自己脑补的流程，直接做成了默认工作流：先判断意图、再分解任务、并行找资料、最后验证结果。说白了，就是把“别急着上锤子”这件事制度化了。

![Oh My OpenCode 官方预览图](https://raw.githubusercontent.com/code-yeongyu/oh-my-openagent/refs/heads/dev/.github/assets/omo.png)

*图：Oh My OpenCode 官方仓库里的预览图。这个项目的气质很鲜明：不是给你多加一个按钮，而是直接给 OpenCode 接上一套更像工程团队的协作外骨骼。*

## OMO 到底是什么

按官方说法，它本质上是围绕 OpenCode 构建的一套 agent harness / orchestration 系统。翻译成人话，大概就是：

- OpenCode 负责提供一个能读代码、改文件、跑命令的工作现场
- OMO 负责把“谁来查、谁来想、谁来做、谁来验”这件事安排明白

所以它不是单纯多装几个 prompt，也不是“把 agent 名字起得更神话一点”这么简单。它更像是给 OpenCode 补了一层真正能调度分工的工作流。

官方现在项目名更常见的是 **Oh My OpenAgent**，但安装包、命令名和很多社区叫法里，依然经常能看到 **oh-my-opencode / OMO**。第一次看到这两个名字混着出现，不用慌，这不是你电脑闹鬼，更不是你装了两个版本的平行宇宙。

## 它更适合哪种人

如果你是下面这类使用习惯，OMO 大概率会比“单 agent 硬怼到底”更对胃口：

- 你本来就长期待在终端、Tmux、SSH、Neovim 这一套环境里。
- 你做的不是“一次只改一个函数”的小修小补，而是经常要查代码、翻文档、跑命令、验证输出一起上。
- 你已经开始觉得，真正拖慢 AI 干活的，不一定是模型不够强，而是流程太乱。
- 你愿意花一点时间把配置和规则理顺，换后面持续更省心。

如果你只想要一个“帮我补全这行代码”的工具，那编辑器补全插件通常更省钱，也更安静。OMO 更适合那种“把任务交给它一段时间，它别半路装睡”的场景。

## 为什么我会觉得它比普通 agent 更能打

最直观的差别，是它把多 agent 协同这件事做得比较像回事。

比如在官方设定里，你能看到像 `Sisyphus`、`Prometheus`、`Atlas`、`Oracle`、`Librarian`、`Explore` 这类角色。名字听起来有点像神话人物团建，但分工其实很务实。为了避免第一次看文档时“这位是谁、那位又管什么”的迷惑，我把它们压成一张更直观的表：

| 角色 | 主要职责 | 更适合用什么模型类型 |
| --- | --- | --- |
| `Sisyphus` | 主编排者。负责理解你的真实意图、拆任务、派 agent、盯验证，有点像不睡觉的技术负责人。 | 适合放在强规划、强推理、上下文处理能力好的旗舰模型上，优先 Claude Opus / Kimi K2.5 / GLM 这类擅长 orchestration 的模型。 |
| `Prometheus` | 规划师。更像先采访你、再写作战方案的那位，不急着写代码。 | 适合放在长上下文、提问和规划能力强的模型上，重点是“会想清楚再动手”。 |
| `Atlas` | 执行者。拿着已经成型的计划去推进实现、调度专门工种、回收结果。 | 适合执行稳定、工具调用成熟的模型，要求不是嘴上会说，而是真能持续干活。 |
| `Oracle` | 架构顾问 / 疑难问题会诊。适合在方向不明确、调试卡住、需要判断取舍时出场。 | 适合高质量推理模型，宁可贵一点，也别在关键架构判断上省错钱。 |
| `Librarian` | 文档和外部代码检索。负责去官方文档、GitHub、示例仓库里把证据找回来。 | 适合速度快、成本低、检索能力强的模型，重点是便宜、稳、能并发。 |
| `Explore` | 仓库内探索。快速 grep、找文件、查模式，像一个永远不嫌麻烦的代码搜索员。 | 同样适合便宜且快的小模型，因为这类活通常量大、频繁、适合并行。 |

这套设计的价值在于，主 agent 不需要一边想架构、一边 grep 全仓库、一边查官方文档、一边假装自己没漏掉边界条件。该查的让查资料的去查，该搜的让搜索 agent 去搜，该拍板的交给更贵但更稳的模型去拍板。人写代码时都知道分工重要，轮到 agent 了，当然也一样。

## 安装 OMO，先别把自己装晕

官方现在其实挺直白，甚至有点“别自己硬装了”的意思。README 里的建议是：**直接把安装文档丢给你手头的 AI agent，让它照着装。**

比如官方给人的版本，基本就是这一句：

```text
Install and configure oh-my-opencode by following the instructions here:
https://raw.githubusercontent.com/code-yeongyu/oh-my-openagent/refs/heads/dev/docs/guide/installation.md
```

这个思路我反而挺认同。OMO 本来就是给 agent 工作流加外骨骼的，你再手动一点点去抄命令、改配置，多少有点让挖掘机回去拿铁锹。只要你手头的 agent 基本靠谱，让它按官方安装文档自动走一遍，通常比自己手抖配错更省事。

如果你还是想自己手动装，官方最直接的命令是：

```bash
bunx oh-my-opencode install
```

装完以后，建议顺手跑一下医生：

```bash
bunx oh-my-opencode doctor
```

这一步很值得做。因为 AI 工具最烦人的问题，往往不是“不会用”，而是“你以为它已经装好了，结果它其实只是在礼貌地失败”。`doctor` 这类自检命令，能帮你少走很多“明明昨天还能跑”的弯路。

如果你已经装好了 OpenCode，安装过程中通常会顺带引导你检查 provider / auth 配置。这里我强烈建议按官方交互流程走，不要把 API key 直接贴进截图里，然后再发到博客上给全世界观摩。热爱分享很好，热爱轮换密钥就大可不必了。

## 第一次上手，最实用的方式不是背 agent 名字

OMO 最出圈的入口基本就是这句：`ultrawork`，也就是 `ulw`。

如果你已经在项目目录里进了 OpenCode，通常可以这么开始：

```bash
cd /path/to/your/project
opencode
```

然后在交互界面里输入：

```text
ultrawork
# 或者更短一点
ulw
```

它背后的意思其实很简单：把高精度模式、自动规划、并行 agent、验证回路这些东西一起拉起来。你不用先把每个角色都背熟，再决定要不要按下开始键。先让它跑一遍，你很快就能知道这套工作流是不是你的菜。

当然，`ulw` 也不是仙丹。它更像“把整支施工队都叫来了”，适合复杂一点、需要多步处理的任务。你如果只是想问一句“这个配置项是干嘛的”，那还是别上来就把整个交响乐团搬进客厅。

## 结合我自己的配置，OMO 为什么比较顺手

我自己现在这套 OpenCode / OMO 配置，主要还是围绕这几个位置：

```bash
~/.config/opencode/
~/.config/opencode/opencode.json
~/.config/opencode/oh-my-opencode.json

~/.local/share/opencode/
~/.local/share/opencode/auth.json
~/.local/share/opencode/opencode.db
~/.local/share/opencode/snapshot/
~/.local/share/opencode/tool-output/
```

这里有个很实际的感受：一旦你把“全局配置放哪、插件配置放哪、运行痕迹放哪”这三件事分清楚，后面很多排错都会轻松很多。

比如我自己的 `opencode.json`，核心思路一直都很朴素：

```jsonc
{
  "$schema": "https://opencode.ai/config.json",
  "plugin": [
    "oh-my-opencode@latest",
    "superpowers@git+https://github.com/obra/superpowers.git"
  ]
}
```

这段配置本身没什么玄学，重点在于它体现了一个事实：OpenCode 很适合继续往下长，而 OMO 刚好就是把“继续往下长”这件事做得更像工程体系，而不是单次脚本。

另外我自己机器上也还有 `oh-my-opencode.json` 这类配置文件。里面最有意思的，不是某个具体字段，而是你会明显看到它已经不是“单模型单人格”那套玩法了，而是开始出现不同 agent、不同 category、不同职责的组合。也就是说，你不是只在调一个 AI，而是在调一套工作流。

官方现在更偏向使用 `oh-my-openagent` 这个名字，但兼容层还保留着，所以你在本机配置里同时看到 legacy 名称和新名称，不用急着怀疑人生。很多时候这只是项目在过渡期里，尽量不把老用户一脚踹下车。

## 我觉得它的优点，确实不只是“看起来厉害”

### 1. 多 Agent 协同是真的有用

这不是为了把面板做得更热闹，而是因为复杂任务本来就不该让一个 agent 同时兼职侦察兵、架构师、施工队和质检员。该并行的时候并行，该分工的时候分工，完成度会稳定很多。

### 2. 很适合终端流工作方式

如果你的工作已经在命令行里发生，OMO 这类增强层会非常自然。你不用在“浏览器里的 AI”和“终端里的真实现场”之间来回切换，思路更完整，动作也更顺。

### 3. 它会逼流程更像工程，而不是像许愿

这一点我挺喜欢。先分析、再规划、再执行、再验证，这些本来就是正常开发流程。OMO 只是把这套纪律写进了系统行为里。懒人会觉得它有点啰嗦，踩过坑的人会觉得它终于像个同事了。

## 它的缺点也很明显，尤其是钱包会先感受到

### 1. 比较费 token

这是最现实的一条。多 agent、深度规划、背景检索、独立验证，这些东西都不是凭空出现的，背后基本都在烧 token。

所以 OMO 的强项和它的成本，几乎是同一枚硬币的正反面：你想要更完整的流程、更像团队协作的效果，那通常就得接受它不像“单次问答”那么省。

### 2. 名字和兼容层对新手稍微有点绕

仓库叫 `oh-my-openagent`，安装包常见的是 `oh-my-opencode`，配置文件里你又可能看到新旧名字同时出现。虽然官方已经尽量兼容，但第一次接触时，确实会有一种“我到底该认哪个名字当大哥”的迷惑感。

### 3. 不适合所有任务都开满编制

如果你只是改一句文案、修一个错字、看一个配置项说明，那上来就 `ulw`，多少有点像为拧灯泡开项目启动会。它能干，不代表每次都值得这么干。

## 我自己的建议：把它当“放大器”，别当“替身”

OMO 最适合的打开方式，不是把自己变成纯旁观者，而是把它当成一个能放大你已有工作流的系统。

你本来就会拆任务、会判断风险、会区分“先查资料”和“直接动手”的差别，那它会非常好用；你如果希望它自动理解一切、自动替你兜住一切、自动在你一句话都没说清楚的前提下把事情办成，那最后大概率还是得你自己回来收拾残局。

换句话说，它不是魔法棒，更像外骨骼。你自己先会走，它能让你走得更快、更稳，还更不容易把腰闪了。

## 最后一句

如果说 OpenCode 解决的是“AI 能不能真的进终端干活”，那 OMO 解决的更像是“它进来以后，到底能不能像个靠谱团队那样干活”。

我的结论目前很简单：它确实更强，也确实更费 token；它确实更像工程化工作流，也确实不太适合“随口问一句就走”的轻量场景。

但如果你已经开始认真把 AI 当生产工具，而不只是聊天窗口，那 OMO 这条路，至少很值得你亲手走一遍。
